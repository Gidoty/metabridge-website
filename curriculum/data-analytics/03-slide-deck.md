## SLIDE 26 — CTEs — WRITE READABLE SQL

**Visual:** Two code blocks side by side. Left: a messy nested subquery — 25 lines, hard to read, highlighted in red. Right: the same logic rewritten as a CTE — modular, readable, highlighted in green.

**Content:**
```
CTEs — COMMON TABLE EXPRESSIONS

The Problem:                    The Solution:
Nested subqueries               Named, reusable query blocks
are unreadable                  

SELECT * FROM                   WITH clean_customers AS (
(SELECT customer_id,                SELECT customer_id,
        SUM(amount)                         SUM(amount) AS total
 FROM (SELECT * FROM t              FROM transactions
       WHERE date > '2024')         WHERE date > '2024'
 GROUP BY customer_id)              GROUP BY customer_id
WHERE total > 100000            )
                                SELECT * FROM clean_customers
                                WHERE total > 100000

SAME RESULT. Very different readability.

Rule: If your subquery runs more than 2 levels deep, use a CTE.
```

**Speaker notes:** CTEs are one of those things that seem advanced but are actually simpler than alternatives. Once students write a CTE they never want to go back to nested subqueries.

## SLIDE 40 — CORRELATION IS NOT CAUSATION

**Visual:** Three scatter plots. Left: spurious correlation — ice cream sales vs drowning rates, strong R². Middle: correlation with confound labelled "Hot Weather" pointing to both variables. Right: the same data, but with the confound removed — weak true relationship.

**Content:**
```
THE MOST IMPORTANT STATISTICAL PRINCIPLE

ICE CREAM SALES       DROWNING INCIDENTS
    correlate with → in Nigerian coastal states
                   R² = 0.74 (strong correlation!)

Does ice cream cause drowning? NO.
Hot weather causes BOTH.

TO ESTABLISH CAUSATION YOU NEED:
1. Correlation (the numbers move together)
2. Time order (cause must come before effect)
3. Elimination of alternatives (rule out confounds)
```

## SLIDE 41 — HYPOTHESIS TESTING — SIMPLIFIED

**Visual:** A/B test scenario. Left: Grey button "Original" with conversion rate 3.2%. Right: Green button "Variation" with conversion rate 3.8%. Below: p-value display "p = 0.02" with a green checkmark and "Statistically significant — launch the green button."

**Content:**
```
HYPOTHESIS TESTING — IS IT REAL OR RANDOM?

Scenario: We changed the app button from grey to green.
Conversion went from 3.2% → 3.8%.
Is this real? Or just luck?

H₀ (Null): Button colour has no effect. Difference is random.
H₁ (Alternative): Green button genuinely converts better.

Run a t-test. Result: p = 0.02

p < 0.05 = Less than 5% chance this is random
→ REJECT H₀ → The green button really works → Launch it

p > 0.05 = Could be random → Don't launch yet → Get more data

⚠  STATISTICAL SIGNIFICANCE ≠ PRACTICAL SIGNIFICANCE
    p = 0.001 and a 0.01% improvement = significant but useless
    Business case matters as much as the p-value.
```

**Speaker notes:** This is how every tech company makes product decisions. Show students this framework — it's used at Paystack, Konga, MTN — every time they A/B test anything.
